1 · Kubernetes DNS

Kubernetes에서 CoreDNS의 역할

CoreDNS는 Kubernetes 클러스터 내부의 DNS resolver입니다. Pod는 kubelet이 구성한 /etc/resolv.conf를 통해 CoreDNS Service ClusterIP로 DNS query를 보냅니다.

kube-dns 이름

구현은 CoreDNS지만, KubeDNS 호환성을 위해 Service 이름은 보통 kube-dns입니다.

Pod가 시작할 때 들어가는 resolver 설정

search <namespace>.svc.cluster.local svc.cluster.local cluster.local
nameserver <CoreDNS Service ClusterIP>
options ndots:5

Pod는 이 설정을 보고 CoreDNS Service ClusterIP로 DNS query를 보냅니다. 이 파일은 뒤에서 다룰 ndots query 증폭의 출발점이기도 합니다.

kubeletPod DNS policy와 cluster DNS 설정을 바탕으로 resolv.conf 구성
Pod resolvernameserver/search/ndots를 읽고 query 생성
kube-dns SVCCoreDNS Pod로 DNS 트래픽 분산

CoreDNS의 3가지 역할

Service name resolving

Service 이름 또는 FQDN을 ClusterIP로 resolve합니다. CoreDNS는 API server를 watch하여 Service, EndpointSlice, Namespace, Pod 정보를 동기화합니다.

Reverse DNS

Pod IP나 Service ClusterIP에 대해 Kubernetes DNS name을 PTR record로 반환합니다. 네트워크 로그에 IP만 남았을 때 유용합니다.

Non-cluster forwarding

Kubernetes zone 밖이고 cache hit가 아닌 FQDN은 Corefile의 forward 경로를 통해 upstream resolver로 전달합니다. 

dig -x 10.96.20.30
# connection log에 찍힌 IP가 어떤 Kubernetes DNS name인지 확인할 때 사용
2-1 · Plugin-chain

CoreDNS는 plugin-chain으로 query를 처리한다

CoreDNS의 기능은 대부분 plugin으로 구성됩니다. 각 plugin은 query를 보고 직접 응답할지, 다음 plugin으로 넘길지, 또는 upstream으로 전달할지를 결정합니다.

CoreDNS is powered by plugins

Kubernetes service discovery, cache, forwarding, metrics, logging이 모두 plugin 단위로 붙습니다.

Plugin-chain 처리 모델

DNS query CoreDNS server block으로 들어오는 요청
compiled order Corefile 순서가 아니라 plugin.cfg 기준 순서
plugin decision 응답하거나, 다음 plugin 호출하거나, fallthrough
backend work cache, Kubernetes API cache, upstream resolver 등
DNS response A/PTR/NXDOMAIN/SERVFAIL 등으로 반환
응답하고 chain 종료 cache hit 또는 kubernetes plugin이 authoritative answer를 가진 경우
다음 plugin 호출 현재 plugin이 query를 처리하지 않고 다음 단계로 넘기는 경우
끝까지 응답 없음 어떤 plugin도 response를 만들지 못하면 SERVFAIL

plugin.cfg 기준 실행 순서 예시: AKS Corefile 주요 plugin 발췌

reload Corefile 변경 감지와 hot reload
ready / health readiness와 liveness endpoint
prometheus DNS request metric 수집
errors / log 오류와 query log 기록
loadbalance 응답 record 순서 shuffle
cache 응답 cache hit/miss 처리
...
template 패턴 기반 synthetic response
kubernetes cluster.local authoritative answer
loop forwarding loop 감지
forward cluster 외부 query를 upstream으로 전달
Corefile의 나열 순서가 아니라 CoreDNS에 컴파일된 plugin.cfg 순서가 기준입니다. 위 diagram은 AKS 기본 Corefile에 등장하는 주요 plugin만 발췌한 축약 흐름입니다.

Corefile은 “구성”이고 실행 순서는 별도다

Corefile은 어떤 plugin을 사용할지와 각 plugin의 argument를 보여줍니다. 하지만 Corefile에 적힌 나열 순서가 그대로 실행 순서가 되는 것은 아닙니다. 실제 순서는 CoreDNS가 컴파일될 때 포함된 plugin.cfg에 의해 고정됩니다.

운영 관점에서는 “Corefile에 어떤 plugin이 켜져 있는가”와 “plugin chain에서 어떤 plugin이 응답을 만들었는가”를 구분해서 봐야 합니다.

AKS에서 특히 봐야 하는 plugin 그룹

kubernetes

cluster.local zone의 authoritative answer.

cache

TTL 동안 응답을 메모리에 저장.

forward

cluster 밖 query를 upstream resolver로 전달.

prometheus/log

관측과 디버깅을 위한 노출 지점.

External plugin도 존재하지만, AKS에서는 built-in plugin 외 external plugin 사용이 제한됩니다. 그래서 AKS 운영에서는 기본 제공 plugin을 어떻게 구성하고 관측할지가 더 중요합니다.
2-2 · Go Concurrency

Go runtime은 많은 DNS query를 가볍게 병렬 처리한다

CoreDNS는 Go로 작성되어 있고, DNS 요청 처리는 goroutine 기반 concurrency 모델과 잘 맞습니다. 요청마다 무거운 OS thread를 새로 만드는 구조가 아닙니다.

핵심 모델

Go runtime이 goroutine을 OS thread 위에 스케줄링하고, I/O 대기 중에도 다른 query 처리가 계속 진행됩니다.

Goroutine 기반 DNS query 처리

CoreDNS request goroutines
G1Service name lookup
G2cache lookup
G3upstream forward 대기
G4reverse DNS lookup
하나의 query가 upstream I/O로 대기해도 다른 goroutine의 query는 계속 처리될 수 있습니다.
OS threads / CPU cores
Thread M1Go scheduler가 goroutine을 배치
Thread M2CPU core 수에 따라 병렬 처리
Thread M3I/O 대기와 실행 작업을 교차 처리
CoreDNS가 DNS 요청마다 OS thread를 새로 만드는 모델이 아닙니다.
Go runtime이 goroutine을 OS thread 위에 스케줄링하는 흐름.

CoreDNS workload와 잘 맞는 이유

  • DNS 요청은 CPU 연산보다 메모리 조회, cache 조회, upstream 질의 같은 I/O 성격이 많습니다.
  • 각 goroutine은 독립적으로 plugin chain을 따라가며 DNS response를 만들 수 있습니다.
  • 하나의 요청이 I/O로 대기하는 동안에도 다른 요청은 계속 처리될 수 있습니다.
  • 요청이 많아질수록 CPU core 수에 따라 병렬 처리 효과가 나타날 수 있습니다.
plugin 작성/운영 관점

한 plugin 인스턴스가 여러 goroutine에 의해 동시에 호출될 수 있으므로 plugin 코드는 동시에 호출된다는 가정 아래 thread-safe하게 동작해야 합니다.

3 · Corefile and AKS ConfigMaps

Corefile과 AKS 구성

AKS에서는 기본 coredns ConfigMap을 직접 수정하기보다 coredns-custom ConfigMap을 통해 기본 Corefile에 설정을 주입하는 방식으로 이해하는 것이 안전합니다.

관여 파일

/etc/coredns와 /etc/coredns/custom에 각각 coredns, coredns-custom ConfigMap이 mount됩니다.

Corefile 서버 블록 문법

[SCHEME://]ZONE [:PORT] {
    PLUGIN [ARGUMENTS]
}

ZONE

서버 블록이 담당할 DNS 이름 영역입니다. .은 DNS root zone이므로 사실상 모든 DNS 질의를 의미합니다.

SCHEME

DNS 요청을 받을 프로토콜입니다. 생략 시 UDP/TCP 기반 DNS이며, dns://, https:// 형태도 가능합니다.

PORT

listen 포트입니다. 기본값은 53이지만 일반적으로 .:53처럼 명시합니다.

AKS 기본 CoreDNS Corefile 예시

# AKS의 Default CoreDNS Corefile
.:53 {
        errors
        ready
        health {
          lameduck 5s
        }
        kubernetes cluster.local in-addr.arpa ip6.arpa {
          pods insecure
          fallthrough in-addr.arpa ip6.arpa
          ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf
        cache 30
        loop
        reload
        loadbalance
        import custom/*.override
        template ANY ANY internal.cloudapp.net {
          match "^(?:[^.]+\.){4,}internal\.cloudapp\.net\.$"
          rcode NXDOMAIN
          fallthrough
        }
        template ANY ANY reddog.microsoft.com {
          rcode NXDOMAIN
        }
    }
      import custom/*.server
직접 수정 주의

coredns ConfigMap을 직접 수정하면 AKS Control Plane의 reconcile 대상이 될 수 있습니다. 기본 Corefile 변경 대신 coredns-custom을 통한 주입 구조를 사용합니다.

3 · Plugins

주요 plugin과 실행 순서

Corefile은 어떤 plugin을 사용할지 보여주지만, 실제 plugin 실행 순서는 Corefile에 적힌 순서가 아니라 CoreDNS의 plugin.cfg에 의해 컴파일 시 고정됩니다.

SERVFAIL

plugin chain 끝까지 response가 만들어지지 않으면 SERVFAIL이 반환됩니다.

kubernetes

cluster.local과 cluster domain에 대한 authoritative DNS 데이터를 제공하고, API를 watch하여 Service/EndpointSlice/Pod/Namespace cache를 유지합니다.

forward

cluster 외부 도메인을 upstream DNS 서버에 recursive query하여 non-authoritative 응답을 반환합니다. UDP, TCP, DNS-over-TLS를 지원합니다.

cache

응답 캐싱을 수행합니다. 기본 설정에서는 약 9984개 엔트리와 최대 30초 캐시를 언급합니다.

loop

CoreDNS → systemd-resolved → back to CoreDNS 같은 DNS forwarding loop를 감지하고 서버를 정지시킵니다.

reload

약 30초마다 Corefile SHA512 해시를 점검하고, 변경 시 서버 재시작 없이 새로운 plugin chain을 로드합니다.

health / ready

/health는 liveness, /ready는 plugin 준비 상태를 노출합니다. 하나라도 준비되지 않으면 ready는 503을 반환합니다.

prometheus

:9153/metrics로 CoreDNS와 plugin metric을 Prometheus 형식으로 노출합니다.

errors / log

errors는 error response를 stdout에 출력하고, log는 query name, response code, duration 등을 기록합니다. log는 상시 활성화 시 오버헤드가 있습니다.

hosts

Linux /etc/hosts와 같은 형식의 정적 IP-domain 매핑을 CoreDNS를 통해 cluster 차원에서 제공합니다.

plugin 목록 확인

kubectl -n kube-system exec deployment/coredns -c coredns -- \
  coredns -plugins | tr ' ' '\n' | sort
실행 순서

Corefile에서 plugin을 나열한 순서는 실행 순서에 영향을 주지 않습니다. CoreDNS plugin.cfg에 정의된 순서가 컴파일 시 고정됩니다.

4 · Query Flow

K8s 내부 DNS query flow

DNS query 수가 늘어나는 지점과 실제 resolution이 일어나는 지점을 분리해서 봅니다. 특히 ndots는 CoreDNS 내부가 아니라 Pod resolver 단계에서 영향을 줍니다.

중요한 오해

Container 안에 별도 DNS 서버가 떠 있거나 Linux OS가 recursive DNS 서버 역할을 하는 것이 보통은 아닙니다.

Application직접 packet을 만들기보다 runtime/libc resolver 호출
→
ResolverGo net.Resolver, JVM cache, socket.getaddrinfo, libuv/c-ares 등
→
resolv.confnameserver / search / ndots 확인
→
kube-dns SVCClusterIP:53
→
CoreDNS Podplugin chain 실행
→
Responsekubernetes plugin 또는 forward/upstream 결과

Pod resolv.conf는 누가 만드는가

spec:
  dnsPolicy: ClusterFirst

ClusterFirst에서는 Pod의 nameserver가 node의 일반 DNS가 아니라 kube-dns Service IP로 설정됩니다.

흐름을 바꾸는 설정

  • dnsPolicy: Default: node DNS 설정 사용
  • dnsPolicy: None: 사용자가 dnsConfig 지정
  • hostNetwork: true: 기본 DNS policy가 달라질 수 있음
  • dnsConfig.options.ndots: Pod 단위 ndots override

kube-dns Service에서 CoreDNS Pod까지

kubectl -n kube-system get service kube-dns
kubectl -n kube-system get endpointslice -l kubernetes.io/service-name=kube-dns
kubectl -n kube-system get pods -l k8s-app=kube-dns -o wide
Pod
  -> kube-dns Service ClusterIP:53
  -> kube-proxy / dataplane rules
  -> CoreDNS Pod IP:53

CoreDNS 내부 처리 분기

Service name 조회

backend.default.svc.cluster.local.
  -> kubernetes plugin
  -> Service / EndpointSlice cache 조회
  -> ClusterIP 또는 endpoint record 응답

외부 도메인 조회

www.microsoft.com.
  -> kubernetes plugin zone 밖
  -> cache miss면 forward plugin
  -> upstream resolver
  -> response cache 저장 후 반환

ndots가 끼어드는 위치

ndots는 CoreDNS 내부 동작이 아니라 Pod resolver 단계에서 영향을 줍니다. ndots:5 상태에서 api.github.com을 조회하면 resolver가 search suffix 후보를 먼저 만들 수 있습니다. 

api.github.com.<namespace>.svc.cluster.local.
api.github.com.svc.cluster.local.
api.github.com.cluster.local.
api.github.com.
구분

CoreDNS가 recursive하게 이름을 바꿔 다시 질의하는 것이 아닙니다. Pod resolver가 여러 후보 query를 만들고 CoreDNS는 여러 개의 독립 query를 받습니다.

5 · Metric, Log, Trace

CoreDNS 관측: 보이는 것과 안 보이는 것

CoreDNS는 prometheus, log, trace plugin을 갖고 있지만 AKS와 Azure Monitor에서는 기본 수집 설정의 경계를 같이 봐야 합니다.

핵심

plugin이 존재하는 것과 Azure Monitor에 자동으로 수집되는 것은 다른 문제입니다.

Metric

prometheus plugin은 CoreDNS metrics를 9153 포트로 노출합니다. 

.:53 {
  ...
  prometheus :9153
  ...
}

Azure Managed Prometheus는 기본적으로 CoreDNS scraping을 하지 않으므로, ama-metrics-settings-configmap 설정이 필요합니다.

ama-metrics Deployment에 ama-metrics-settings-configmap이 mount된 예시. 
apiVersion: v1
kind: ConfigMap
metadata:
  name: ama-metrics-settings-configmap
  namespace: kube-system
data:
  cluster-metrics: |-
    default-targets-scrape-enabled: |-
      coredns = true
    minimal-ingestion-profile: |-
      enabled = true
  config-version: ver1
  controlplane-metrics: |-
    minimal-ingestion-profile: |-
      enabled = true
  prometheus-collector-settings: |-
    cluster_alias = ""
    debug-mode = false
    https_config = true
    secrets_access_namespaces = ""
  schema-version: v2

Log

log, errors plugin은 DNS traffic과 오류 상태를 기록할 수 있습니다. 기본 Corefile에는 log plugin이 빠져 있으므로 coredns-custom에 override를 추가하고 restart합니다. 

data:
  log.override: |
    log
ama-logs-rs Deployment에 container-azm-ms-agentconfig가 mount된 예시.
수집 경계

log plugin을 켜도 Container Insights 기본 설정만으로는 Log Analytics Workspace에 CoreDNS 로그가 수집되지 않을 수 있습니다. kube-system 같은 system namespace container logs가 기본 제외되기 때문입니다. 

apiVersion: v1
kind: ConfigMap
metadata:
  name: container-azm-ms-agentconfig
  namespace: kube-system
data:
  schema-version: "v1"
  config-version: "1.0.0"
  log-data-collection-settings: |-
    [log_collection_settings]
      [log_collection_settings.stdout]
        enabled = true
        collect_system_pod_logs = ["kube-system:coredns"]

      [log_collection_settings.stderr]
        enabled = true
        collect_system_pod_logs = ["kube-system:coredns"]
설정 배포 후 ama-log pod가 rollout restart되는 흐름. 자동 restart가 없으면 직접 restart를 수행합니다.

Trace

기본 상태

AKS CoreDNS 바이너리에 trace plugin은 존재하지만 기본 설정에는 활성화되어 있지 않습니다. coredns-custom을 통해 trace plugin을 설정해야 합니다.

지원 endpoint

trace telemetry endpoint type은 zipkin과 datadog만 지원하며, OpenTelemetry가 아닌 OpenTracing 기반입니다.

trace plugin 활성화 흐름

CoreDNS trace는 기본적으로 켜져 있지 않으므로, Zipkin endpoint를 준비한 뒤 coredns-custom에 trace.override를 추가하고 CoreDNS를 재시작해서 적용합니다. 테스트 구간에서는 sampling 값인 every를 작게 잡고, 공유 클러스터에서는 더 크게 잡아 trace 양을 제한합니다. 

trace zipkin http://<zipkin-cluster-ip>:9411/api/v2/spans {
        every <TRACE_EVERY>
        service aks-coredns
}
endpoint는 가능하면 IP로 지정

CoreDNS process가 Zipkin Service 이름을 다시 CoreDNS로 질의하는 상황을 줄이기 위해, trace endpoint는 Service DNS 이름보다 Zipkin Service ClusterIP를 사용하는 편이 안전합니다. 

kubectl apply -f spec/03-dns-tools.yaml
kubectl -n trace-lab rollout status deployment/dns-tools --timeout=5m

kubectl -n trace-lab exec deployment/dns-tools -- sh -c '
for i in $(seq 1 30); do
  dig +short kubernetes.default.svc.cluster.local
  dig +short www.microsoft.com
  dig +short api.github.com
done
'
kubectl -n "$LAB_NS" port-forward service/zipkin 9411:9411
Zipkin으로 CoreDNS trace를 확인한 결과 화면.

Application trace context가 DNS query까지 전파되는지 확인

Application trace와의 correlation

CoreDNS trace plugin은 Zipkin OpenTracing wrapper 기반이라 기본 propagation이 W3C traceparent가 아닙니다. 일반 UDP/TCP DNS query에는 HTTP header처럼 trace context를 실을 carrier도 없습니다. 그래서 CoreDNS servedns span은 application span의 child가 아니라 별도 root trace로 생성됩니다.

traceparent 직접 주입 실험

HTTP 요청에 traceparent header를 직접 넣으면 application trace는 그 trace ID를 사용합니다. 하지만 DNS packet에는 그 header가 실리지 않으므로 CoreDNS trace는 여전히 별도 trace ID를 사용합니다.

현실적인 correlation 기준

자동 parent-child 관계 대신 timestamp, query name, client Pod IP, namespace/workload metadata를 기준으로 application request와 CoreDNS query를 맞춰 보는 방식이 현실적입니다.

DNS protocol-level trace context는 가능한가?

DNS에는 HTTP header 같은 범용 metadata 영역은 없지만, EDNS(0)의 OPT pseudo-RR option을 carrier로 삼는 방식은 이론적으로 가능합니다. EDNS TRACEPARENT draft는 W3C Trace Context의 trace-id, parent-id, trace-flags를 DNS option data에 싣는 방향입니다.

질문 판단
DNS packet에 trace context를 실을 확장 지점이 있는가? EDNS option을 사용하면 기술적으로 가능하다.
일반 AKS Pod → CoreDNS 경로에서 자동으로 전파되는가? 아니다. 일반 resolver가 EDNS trace context를 넣지 않는다.
AKS managed CoreDNS에서 trace plugin 코드를 바꿀 수 있는가? 어렵다. coredns-custom은 Corefile 설정만 바꾼다.
같은 backend로 모으면 하나의 trace tree가 되는가? 아니다. trace ID가 다르면 backend가 자동 병합하지 않는다.
대안: application DNS span과 eBPF/OBI correlation

Application instrumentation은 현재 active span 아래에 dns.resolve 같은 span을 만들 수 있습니다. 이 방식은 application 요청 안에서 DNS lookup 시간이 병목인지 보기 좋지만, CoreDNS 내부 plugin/cache/upstream 경로는 보이지 않습니다. 반대로 eBPF/OBI는 실제 DNS packet과 socket/flow를 관측해 qname, rcode, latency, source workload를 볼 수 있지만, DNS packet에 trace context를 주입하는 방식은 아닙니다. 따라서 가능한 경우 span link 또는 metadata correlation으로 연결하는 접근에 가깝습니다.

6 · Troubleshooting Scenarios

시나리오 기반 설정과 판단

다섯 가지 운영 시나리오를 다룹니다: conditional forward, latency/timeout, ndots query 증가, TTL/cache/prefetch, Pod 내부 DNS 기본 진단.

분류 기준

CoreDNS 자체 문제인지, upstream 문제인지, client resolver가 query를 늘린 것인지 먼저 나눕니다.

1. 특정 도메인을 별도 DNS 서버로 conditional forward

사내 DNS zone을 AKS 기본 upstream이 아니라 별도 DNS 서버로 보내고 싶은 경우입니다. 예시는 corp.example.com을 사내 DNS 서버 10.10.0.10, 10.10.0.11로 forward합니다.

CoreDNS custom 방식

coredns-custom에 .server로 끝나는 key를 추가하여 AKS 기본 Corefile의 import custom/*.server에 의해 로드되게 합니다.

Azure DNS layer 방식

Azure Private DNS Zone 또는 Azure DNS Private Resolver outbound endpoint와 DNS forwarding ruleset을 사용합니다. 여러 VNet/클러스터에서 공유할 때 중앙집중적입니다. 

apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns-custom
  namespace: kube-system
data:
  corp-example-com.server: |
    corp.example.com:53 {
      errors
      forward . 10.10.0.10 10.10.0.11
    }
  • forward 대상은 DNS 이름보다 IP가 안전합니다. upstream 주소를 resolve하기 위해 다시 DNS가 필요한 bootstrap 문제를 피합니다.
  • AKS subnet에서 사내 DNS 서버까지 UDP/TCP 53이 열려 있어야 합니다.
  • 전체 DNS를 사내 DNS로 보내는 것과 특정 zone만 보내는 것은 다릅니다. 
kubectl -n kube-system patch configmap coredns-custom \
  --type json \
  -p='[{"op":"remove","path":"/data/corp-example-com.server"}]'

kubectl -n kube-system rollout restart deployment/coredns
kubectl -n kube-system rollout status deployment/coredns --timeout=5m

2. DNS 응답 지연 또는 timeout

CoreDNS 자체 처리량 부족

높은 QPS, CPU throttling, cache miss 증가, 과도한 NXDOMAIN 등이 원인일 수 있습니다.

Upstream 지연

외부 도메인 forward, upstream resolver 지연, 네트워크 이슈가 원인일 수 있습니다.

1차 대응

CoreDNS는 stateless DNS 서버이므로 일반적으로 CPU limit을 먼저 키우기보다 replica 수를 늘려 수평 확장하는 방식이 흔한 대응입니다.

AKS CoreDNS autoscaling 구조

kubectl -n kube-system get deployment coredns coredns-autoscaler
kubectl -n kube-system get configmap coredns-autoscaler -o yaml
kubectl -n kube-system get deployment coredns-autoscaler -o yaml
coredns-autoscaler Pod
  -> Kubernetes API에서 schedulable node/core 수 확인
  -> coredns-autoscaler ConfigMap의 scaling rule 확인
  -> deployment/coredns replica 수 조정
  -> Deployment controller가 CoreDNS Pod 수 유지
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns-autoscaler
  namespace: kube-system
data:
  ladder: '{"coresToReplicas":[[1,2],[512,3],[1024,4],[2048,5]],"nodesToReplicas":[[1,2],[8,3],[16,4],[32,5]]}'
중요

coredns-autoscaler는 Kubernetes HPA가 아니라 cluster-proportional-autoscaler입니다. CPU 사용률이나 DNS latency를 직접 보고 scale out하는 closed-loop autoscaling이 아니라 node/core 수에 비례해 replica를 계산합니다.

관찰 지표

  • coredns_dns_requests_total
  • coredns_dns_request_duration_seconds
  • coredns_dns_responses_total에서 NXDOMAIN/SERVFAIL 증가
  • coredns_forward_requests_total
  • coredns_forward_request_duration_seconds
  • coredns_cache_hits_total, coredns_cache_requests_total
  • CoreDNS Pod CPU usage / throttling / restart count

NodeLocal DNSCache나 LocalDNS는 Pod → CoreDNS 중앙 경로의 부하와 latency를 줄이는 보조 대안으로, DNS QPS가 매우 높거나 node-local cache hit 이점이 큰 경우 별도 검토합니다.

3. ndots로 인해 DNS query가 불필요하게 늘어나는 경우

search <namespace>.svc.cluster.local svc.cluster.local cluster.local
nameserver <CoreDNS Service ClusterIP>
options ndots:5

ndots:5는 조회하려는 이름의 dot 개수가 5개보다 적으면 resolver가 먼저 search suffix를 붙여 조회한다는 의미입니다.

왜 기본값이 ndots:5인가

Kubernetes Service의 SRV record는 예를 들어 _http._tcp.my-service.default.svc처럼 점이 여러 개 포함될 수 있습니다. ndots를 너무 낮게 잡으면 이런 이름을 search domain 확장 전에 absolute name처럼 먼저 처리할 수 있어, 내부 service discovery 패턴과 어긋날 수 있습니다. ndots:5는 내부 DNS 편의성을 위한 기본값이지만, 외부 도메인을 많이 조회하는 workload에서는 query 증폭을 만들 수 있습니다.

핵심 현상

  • 애플리케이션은 하나의 이름만 조회했다고 생각합니다.
  • Pod resolver는 search suffix 후보를 여러 개 만듭니다.
  • CoreDNS는 여러 독립 query를 받은 것으로 기록합니다.
  • 존재하지 않는 후보는 대부분 NXDOMAIN으로 끝납니다.

중요한 구분

이것은 client-side search expansion입니다. CoreDNS가 이름을 바꿔가며 recursive하게 다시 질의하는 것이 아닙니다. 

application
  -> libc / resolver
  -> /etc/resolv.conf 확인
  -> search suffix 후보 이름 생성
  -> CoreDNS에 후보 query를 하나씩 전송
  -> NXDOMAIN이면 다음 후보 query 전송

Lab에서 확인한 예시

외부 도메인에서도 같은 패턴이 나타납니다. 예를 들어 api.github.com은 점이 2개뿐이므로 ndots:5 기준에서는 search suffix 후보가 먼저 생성될 수 있습니다. 

api.github.com.<namespace>.svc.cluster.local.
api.github.com.svc.cluster.local.
api.github.com.cluster.local.
api.github.com.<cloud-internal-domain>.
api.github.com.
query explosion

앞의 후보들은 대부분 NXDOMAIN으로 끝나고 마지막 absolute query에서 성공합니다. 외부 hostname lookup이 초당 100번 발생하고, 각 lookup마다 성공 전 4개의 search suffix query가 추가되면 초당 400개 이상의 불필요한 DNS query가 생길 수 있습니다. 

ndots-target.ndots-lab.svc.cluster.local.ndots-lab.svc.cluster.local.
ndots-target.ndots-lab.svc.cluster.local.svc.cluster.local.
ndots-target.ndots-lab.svc.cluster.local.cluster.local.
ndots-target.ndots-lab.svc.cluster.local.<azure-internal-domain>.
ndots-target.ndots-lab.svc.cluster.local.
Lab 결론
  • ndots:5 + trailing dot 없음: search suffix query가 발생합니다.
  • trailing dot 있음: 실제 Service FQDN만 조회됩니다.
  • ndots:1 + trailing dot 없음: 실제 Service FQDN만 조회됩니다.

완화 방법

  • 외부 도메인 또는 긴 FQDN을 자주 조회하는 Pod에는 dnsConfig.options.ndots 값을 낮추는 방식을 검토합니다.
  • trailing dot을 붙이면 search suffix 확장을 건너뛸 수 있으나 애플리케이션과 TLS hostname 처리 방식을 확인해야 합니다.
  • CoreDNS log, coredns_dns_requests_total, coredns_dns_responses_total, NXDOMAIN 비율, forward latency를 함께 봅니다.
  • DNS query가 많은 워크로드에서는 connection reuse, keep-alive, application-level DNS cache도 점검합니다.

Pod 단위 ndots 조정

ndots:2로 낮추면 api.github.com처럼 점이 2개 이상인 외부 도메인은 처음부터 absolute name으로 먼저 조회되어 query 증폭이 줄어듭니다.

trade-off

내부 service 이름이나 SRV record lookup이 search domain 확장에 의존한다면 동작 차이가 생길 수 있습니다. cluster-wide 변경보다 외부 API 호출이 많은 특정 Pod에서 제한적으로 검증하는 편이 안전합니다. 

spec:
  dnsPolicy: ClusterFirst
  dnsConfig:
    options:
      - name: ndots
        value: "2"

4. TTL 만료 시 upstream resolution latency 줄이기

CoreDNS cache plugin은 upstream, database, Kubernetes API 등 backend 조회 비용이 있는 응답을 일정 시간 메모리에 저장합니다. AKS 기본 Corefile에는 보통 cache 30이 포함되어 최대 30초 캐시로 동작합니다.

왜 latency가 생기나

cache hit이면 메모리에서 바로 응답하지만, miss 또는 TTL 만료 시 backend/upstream resolver로 다시 질의합니다. 외부 도메인은 forward plugin 왕복 시간이 붙을 수 있습니다.

짧은 TTL + 높은 QPS

인기 hostname의 TTL이 짧고 QPS가 높으면 TTL 만료 시점마다 여러 client가 동시에 upstream resolution을 유발할 수 있습니다.

prefetch

cache {
          prefetch AMOUNT [DURATION] [PERCENTAGE%]
}
  • AMOUNT: popular query 판단을 위한 최소 query 횟수
  • DURATION: query 사이 gap 기준. 기본값은 1m
  • PERCENTAGE: TTL이 이 비율 아래로 떨어지면 prefetch 수행. 기본값은 10%
  • prefetch 5 1m 10%: TTL 하위 10% 시점에 미리 갱신

관찰 지표

  • coredns_cache_hits_total
  • coredns_cache_requests_total
  • coredns_cache_prefetch_total
  • coredns_forward_request_duration_seconds
  • coredns_forward_requests_total

Wrap-up

운영자가 가져갈 판단 프레임

CoreDNS 문제는 Pod resolver, kube-dns Service, CoreDNS plugin chain, upstream resolver를 단계별로 분리해서 보는 문제입니다.

한 문장

CoreDNS가 느린지, upstream이 느린지, resolver가 query를 늘렸는지 먼저 분리합니다.

1. resolver 단계

Pod /etc/resolv.conf, dnsPolicy, dnsConfig.options.ndots, runtime resolver/cache를 확인합니다.

2. Service 경로

kube-dns Service, EndpointSlice, CoreDNS Pod 상태와 분산 경로를 확인합니다.

3. CoreDNS 내부

plugin chain, cache hit/miss, kubernetes zone 여부, forward 여부, plugin 실행 순서 오해를 정리합니다.

4. Observability

Prometheus scraping, Container Insights system namespace log collection, trace plugin 한계를 설정과 함께 봅니다.

5. Scaling

CoreDNS replica scaling은 흔한 1차 대응이지만, AKS autoscaler는 DNS QPS가 아니라 node/core 기반입니다.

6. Query reduction

ndots, trailing dot, connection reuse, app-level DNS cache, CoreDNS cache/prefetch로 query와 latency를 줄입니다.

핵심: DNS 장애는 query 생성 지점과 응답 대기 지점을 분리해서 본다.
